# 人类与AI协同工作流实践 实验

##### 本文旨在沉淀自己在实际项目开发时的操作步骤, 偏个人向, 主要用于备忘和沉淀, 仅供参考, 欢迎友善讨论.

##### 我希望通过这篇文章, 可以帮助你降低决策密度, 消除决策噪点, 更容易掌控vibe coding项目

# 前言

#### 1. 为什么有这篇文章

###### **1.1 目的: 总结可迁移, 可落地的方法论和实操指南**

人机协同编程已经成为主流趋势, 本文旨在探索AI协同在商业化项目中的具体操作流程, 也算是给我自己沉淀一个可执行的步骤.

随着skills的出现, 我们的开发方式也发生变化, 各种五花八门的skills进入我们的开发流程, skills数量的膨胀速度很快, 如何管理skills就成了一个不得不面对的问题--**我们都知道无序发展对一个工程来说意味着什么.**

我预期未来大部分项目的skills会像库一样, 成为项目的一部分, 和代码一样重要的资产.

所以如何管理和设计skills就成了一个问题, 如果在项目引入skills初期不进行规划, 混乱难以避免.

我的原则就是通过规范整个开发工作流, 来明确skills在项目开发中如何应用, 甚至让agent自我管理skills.

**目的是**: 降低心智负担 -- AI开发让我们从代码开发中抽离出来, 更专注于逻辑与架构, 这些内容更费脑, 其实大家都发现了, 虽然有AI协同开发, 效率是提升了, 但是人也更累了.

通过结构化整个AI协同流程, **明确skills架构**, 我们可以降低信息管理的负担, 专注做事.


###### 1.2 ⚠️ 降低项目失控的速度

AI协作开发大幅度提升了开发效率, 可以说开发速度变为光速, 但是它也在失控, 大量vibe coding的人可能会有感觉--**项目在逐渐变得不属于你**, 因为项目迭代速度太快, 要关注的东西太多, 很多地方你可能就是review过一次, 就再也不关注它了.

所以, 本文也立志于解决认知负担过重导致项目失控的问题 -- 某些东西让AI去自动做, 没遇到问题前, 你可以安心忽略的那些地方.

这套方法论也是一套脚手架, 让你可以把开发过程中的认知噪点抛开--你只需要配置过一次, 99.9%的时间你可以完全忘记它.

目的就是自动进行**决策降噪**, 我们只关心**重要决策节点**, 细枝末节不再进入决策流.


#### 2. 适合谁, 适合什么场景

适合专业工程师和商业化项目, 对于代码票友来说可能实践意义不大, 也没必要烧这个token.


#### 3. 限制与不足

1. 个人项目经验和吸纳知识速度有限，尽量避免闭门造车，但难免出现有人已经提出成熟想法我又重复造初级轮子的情况.
2. 项目情况不同，导致有些步骤可能对你来说有些落后甚至没有实践价值，或看起来像主观臆想，请无视
3. 很多步骤也偏个人开发习惯, 与个人认知有关

**总结：抛砖引玉**

# 目录

1. 在项目中引入skills
2. 区分 通用skills 与 项目skills, 分别管理
3. skills的类型: 实操步骤, 参考知识库, 文档
4. 第0个skills: install skills
   1. 代码风格/开发习惯学习, 冷启动项目skills
   2. human in loop 处理需要初始化项, 脚手架已经标记ToDo, 人类仅需跟着流程操作完成配置- 比如版本号怎么查, 只有配置了才能开始编码
   3. 废弃install skills
5. 第1个skill: **编码流程skill**, 每一步都对应了一个具体可参考的skills

- 观察与参考现有代码
- 规划
  - 我要做什么
  - 需要哪些步骤
  - 规划(spec)
    - markdown与mermaid
    - 进一步思考: 未来ToDo并分级
  - 执行(编码)
    - human-in-loop: 不懂就问
  - (Optional)测试代码:
    - 为什么是Optional: 不同技术栈, 项目阶段的项目对单测的需求程度不同, 根据需要添加测试代码步骤, 以免造成无必要的心智负担
    - 单测是可以后期补全的, vibe coding项目迭代快, 测试代码生成成本低廉, 废弃速度更快?
  - 二次 ToDo 规划整理
  - (Optional)执行测试代码
  - review
  - (hunman-in-the-loop)hunman review skill (用什么工具?)
  - commit
  - 沉淀开发日志?
  - 自总结更新 **项目skills**

1. 通用skills: 一旦确定内容, 不会频繁改动
   1. 代码规范 skills
   2. commit skills
   3. 观察与参考现有代码 skills
   4. 规划skills(spec, 问题树解析)
   5. review skills
   6. test skills
   7. hunman-in-loop & 沉淀问题知识库
      1. ask
      2. review
   8. ToDo skills: 让agent记录与管理待处理项
   9. 总结skills
   10. 沉淀与迭代 项目skills, 包括但不限于将新引入的 全局变量/方法, 技术栈/三方库 纳入skills
   11. 沉淀开发日志
   12. 更新开发文档, 业务文档
2. 项目skills
   1. 版本号skills
   2. error trace skills
   3. logger skills
   4. 项目架构, 业务流程概述, 需求介绍
   5. 项目原则, 目前项目处于什么阶段, 开发原则与倾向
   6. 文件结构skills: 目录结构, 文件规则
   7. 常用全局变量与全局方法
   8. 常用技术栈与三方库描述
   9. 日志skills: 表达日志记录规则, 统计与沉淀日志具体在哪些业务中被使用
   10. 业务文档: 记录需求迭代变化, 知识库向
   11. 配置与部署操作skills
   12. 问题知识库 skills
